# TUI 10/10 — Modular Refactor + Real Streaming

## Problem

The current Textual TUI in `src/spare_paw/tui/app.py` is a functional MVP with real issues:

1. **Fake streaming.** `router/tool_loop.py:185-191` calls non-streaming `client.chat()`, then retroactively splits the final text by spaces and emits "tokens." Users see a spinner, then a sudden dump. `OpenRouterClient.chat_stream` exists but is never routed through the tool loop.
2. **Latent bug.** `tui/app.py:440` references `self.app_state` (should be `self._app_state`). `/roles` will raise `AttributeError` the first time anyone types it. No test catches it.
3. **Monolithic file.** `tui/app.py` is 659 lines mixing the backend adapter, App shell, command router, image pipeline, history renderer, and remote transport glue.
4. **Duplicated `/help` text** at lines 433 and 623.
5. **Silent error swallowing** in `_load_history_*` (`except Exception: pass`).
6. **Hard-coded 60-char column widths** for message labels (`_format_user_msg`/`_format_bot_label`). Breaks on narrow/wide terminals.
7. **Tool calls render in a hovering panel above input** instead of inline with the conversation.
8. **Missing UX basics:** no input history, no multi-line compose, no chat search, no scroll navigation, no reconnect-on-drop.

## Goal

Claude-Code-style polished streaming chat, built to engineering best practices.

**In scope:**
- Real token streaming end-to-end (router → engine → backend → widget).
- Persistent per-turn `MessageView` widgets with inline collapsible `ToolRow` children.
- Modular split of `tui/app.py` into focused modules under `tui/` and `tui/widgets/`.
- UX basics: input history (↑/↓), multi-line compose (Shift+Enter), scroll navigation (PgUp/PgDn, Home/End, g/G), in-log search (Ctrl+F), copy (mouse + keyboard), responsive layout, explicit error surfacing.
- Reconnect-on-drop for remote mode (SSE).
- Test coverage per module.

**Explicit non-goals (Tier 3 deferred):**
- Telegram live edit-message streaming.
- Conversation switcher / multi-session UI.
- Inline image previews (sixel/kitty).
- Retry-last, edit-previous, theme customization.
- Plugin panes / split views.

## Design

### 1. Architecture (three layers)

**Layer 1 — Router (streaming actually happens).**

- Upgrade `OpenRouterClient.chat_stream` to yield structured `StreamChunk` events, not raw strings.
- Replace `client.chat(...)` inside `run_tool_loop` with a new internal `_stream_and_assemble` helper that consumes `chat_stream`, assembles the final assistant message from deltas, and invokes `on_token` for text deltas as they arrive.
- Delete the fake word-splitter at `tool_loop.py:185-191`.
- Preserve: `asyncio.Semaphore(1)` gating, retries on transient errors, timeouts (`llm_timeout` wraps the whole assemble), token budget circuit breaker, per-tool rate limits, tool execution, stop-signal handling, spawn-agent group_id injection.

**Layer 2 — Engine / Backend protocol.**

- Promote `on_token(str)` and `on_tool_event(ToolEvent)` to the `MessageBackend` protocol, each with a default no-op implementation.
- `core/engine.py` passes `backend.on_token` and `backend.on_tool_event` into `run_tool_loop` on every call.
- Telegram backend: no-op bodies for the new hooks (keeps the existing final-send behavior).
- Webhook backend: `on_token` pushes `{"type":"token","token":...}` SSE events; `on_tool_event` pushes `{"type":"tool_call",...}` events.

**Layer 3 — TUI (module split).**

```
src/spare_paw/tui/
  __init__.py               # re-exports run_tui
  app.py                    # App shell: compose, on_mount, bindings, mode routing
  backend.py                # TUIBackend adapter (MessageBackend impl)
  commands.py               # SlashCommandRouter dispatch table
  events.py                 # Textual Message dataclasses
  streaming.py              # StreamSession (per-turn buffer + token coalescing + finalize)
  theme.py                  # CSS constants + color tokens
  widgets/
    __init__.py
    chat_log.py             # VerticalScroll of MessageView widgets
    message_view.py         # One turn bubble (user or assistant), owns ToolRow children
    tool_row.py             # Collapsible tool call row
    composer.py             # Input with ↑/↓ history + Shift+Enter multiline + slash suggester
    status_bar.py           # Bottom status (connection, model, msg count, tool count)
```

Each file ≤ ~150 LoC. `commands.py` and `streaming.py` are plain Python — unit-testable with no Textual dependency. Widgets use `textual.pilot` for interaction tests.

Public entry unchanged: `from spare_paw.tui import run_tui`.

### 2. Streaming data flow

A single assistant turn:

1. User presses Enter in `Composer` → `App.enqueue_user_turn(text)`.
2. Engine (local) or `RemoteClient` SSE (remote) starts the LLM call.
3. For each token: `TUIBackend.on_token(tok)` → `App.post_message(StreamToken(tok))`.
4. `StreamSession` coalesces tokens over ~16ms windows and calls `MessageView.append` on the active turn widget.
5. On `tool_call_delta` → `TUIBackend.on_tool_event(...)` → `App.post_message(ToolCallStart(...))`.
6. The active `MessageView` pauses text and mounts a `ToolRow` child (status=running).
7. Tool executes (in `run_tool_loop`); on completion `TUIBackend.on_tool_event(end, ...)` → `ToolRow` flips to ✓/✗ with duration and (on error) inline preview.
8. Streaming resumes with subsequent text deltas into the same `MessageView`.
9. On `StreamEnd`: `MessageView.finalize()` swaps the live plain-text buffer for a rendered Markdown widget in place — no scroll jump, no visible "pop."

**Rendering discipline:**

- Plain text during stream; Markdown on finalize. Avoids the broken-code-fence-mid-stream problem.
- One `MessageView` per turn; no global stream buffer. New turn → `ChatLog.mount(MessageView())`.
- History replay uses the same widgets with `mode="historical"` (no streaming, static content). One rendering pipeline for both live and replayed messages.

**Cancellation:** Esc → `App.action_cancel_request()` → cancel the task driving the stream. Active `MessageView` finalizes with what it has plus a dimmed `[Cancelled]` footer. Any still-running `ToolRow` transitions `running → cancelled`.

### 3. Tool call rendering (`ToolRow`)

Collapsed by default:
```
  ▸ read_file(path='src/foo.py')    ✓ 0.3s
  ▸ shell(cmd='pytest tests/')      ✗ 2.1s  Error: ModuleNotFoundError: ...
  ▸ web_search(query='...')         ⣾ running...
```

Expanded (Enter/Space on focused row):
```
  ▾ shell
    args:   { "cmd": "pytest tests/" }
    result: ===== test session starts =====
            ...
            FAILED tests/test_foo.py::test_bar
```

Rules:
- Status icons: `⣾/⣽/⣻/...` animated spinner (running), `✓` success, `✗` error, `⊘` cancelled.
- Duration shown on completion.
- Errors show a one-line preview inline even when collapsed (so failures don't require a click to notice).
- Args truncated to the first ~60 chars on the header line; full args visible in the expanded view.
- Tool rows are children of their owning `MessageView`. Not hovering. Not a separate panel.

### 4. Router changes

`router/openrouter.py`:

```python
@dataclass
class StreamChunk:
    kind: str                              # "text_delta" | "tool_call_delta" | "done"
    content: str | None = None             # text_delta only
    tool_index: int | None = None          # tool_call_delta only
    tool_id: str | None = None             # tool_call_delta only (first chunk per call)
    tool_name: str | None = None           # tool_call_delta only (first chunk per call)
    arguments_fragment: str | None = None  # tool_call_delta only (incremental JSON)
    finish_reason: str | None = None       # done only
    usage: dict | None = None              # done only (if provider includes it)

async def chat_stream(...) -> AsyncIterator[StreamChunk]: ...
```

Parsing: per-SSE-chunk, map OpenRouter fields. `delta.content` → `text_delta`. `delta.tool_calls[i]` → `tool_call_delta` (tool_calls stream incrementally; id/name arrive once, arguments stream in fragments). On `[DONE]` or final chunk with `finish_reason` → emit a `done` chunk.

`router/tool_loop.py`:

```python
async def _stream_and_assemble(client, messages, model, tools, on_token):
    text_parts: list[str] = []
    tool_calls_by_index: dict[int, dict] = {}
    usage: dict = {}
    async for chunk in client.chat_stream(messages, model, tools):
        if chunk.kind == "text_delta":
            text_parts.append(chunk.content)
            if on_token is not None:
                on_token(chunk.content)
        elif chunk.kind == "tool_call_delta":
            tc = tool_calls_by_index.setdefault(chunk.tool_index, {
                "id": "", "type": "function",
                "function": {"name": "", "arguments": ""},
            })
            if chunk.tool_id:
                tc["id"] = chunk.tool_id
            if chunk.tool_name:
                tc["function"]["name"] = chunk.tool_name
            if chunk.arguments_fragment:
                tc["function"]["arguments"] += chunk.arguments_fragment
        elif chunk.kind == "done":
            usage = chunk.usage or {}
    assistant_message: dict = {
        "role": "assistant",
        "content": "".join(text_parts) or None,
    }
    if tool_calls_by_index:
        assistant_message["tool_calls"] = [
            tool_calls_by_index[i] for i in sorted(tool_calls_by_index)
        ]
    return assistant_message, usage
```

Replace the existing `client.chat(messages, model, tools)` call inside `run_tool_loop`. Delete the fake word-splitter block at `tool_loop.py:185-191`. Wrap the whole assemble in `asyncio.wait_for(..., timeout=llm_timeout)` to preserve timeout semantics.

### 6. Backend protocol

`MessageBackend` (currently a `@runtime_checkable` Protocol at `src/spare_paw/backend.py:37`):

```python
class MessageBackend(Protocol):
    async def send_text(self, text: str) -> None: ...
    async def send_file(self, path: str, caption: str = "") -> None: ...
    async def send_typing(self) -> None: ...
    async def send_notification(self, text: str, actions: list[dict] | None = None) -> None: ...
    async def start(self) -> None: ...
    async def stop(self) -> None: ...

    # New optional hooks with default no-op implementations:
    def on_token(self, token: str) -> None: ...
    def on_tool_event(self, event: ToolEvent) -> None: ...
```

Implementations:
- `TUIBackend`: already has these; promote to protocol.
- `TelegramBackend`: no-op.
- `WebhookBackend`: pushes SSE events (verify this already happens; if not, wire it).

### 7. Error handling & surfacing

Every failure path produces a visible, actionable row. No more `except Exception: pass`.

| Failure | Where caught | User sees |
|---|---|---|
| History load fails (local or remote) | `ChatLog.load_history` | Dim-red row: `[!] Could not load recent history: <reason>`; chat still usable |
| Remote auth fail (401) | `RemoteClient` raises `PermissionError` | Banner: `Auth failed — check remote.secret`; input disabled until `/reconnect` |
| Remote SSE drops mid-stream | `App._send_remote` except block | Active `MessageView` finalizes with `[disconnected]` footer; status bar flips red; auto-reconnect begins |
| LLM timeout (`llm_timeout`) | `run_tool_loop` returns error string | Normal assistant message content; `MessageView` renders it |
| Tool timeout / exception | `run_tool_loop` appends error string to tool result | `ToolRow` shows ✗ with inline preview |
| `self.app_state`-class typos | Prevented structurally via explicit injection into `SlashCommandRouter` | Caught at test time |

### 8. Reconnect-on-drop (remote mode)

State machine in `RemoteClient`:

```
connected ──SSE close/error──▶ reconnecting ──backoff (1s, 2s, 4s, …, cap 30s)──▶ connected
                                     │
                                     └──user cancels / /disconnect──▶ disconnected
```

- `App` subscribes to a `ConnectionState` message stream from the client.
- Status bar dot: green (connected), yellow (reconnecting), red (disconnected).
- In-flight response during drop: active `MessageView` finalizes with `[disconnected, will resume]`. After reconnect, the TUI does NOT auto-retry the in-flight prompt — the user re-sends manually. (Automatic retry-last is deferred to Tier 2.)
- Retries unbounded until user cancels, but backoff capped. Every attempt logged.

## Migration / backwards compat

- `MessageBackend.on_token` / `on_tool_event` default to no-ops; existing Telegram and webhook backends keep working without modification.
- Public entry `from spare_paw.tui import run_tui` unchanged.
- No new top-level dependencies (Textual, aiohttp, Rich already present).
- Old `tui/app.py` is fully replaced — no compatibility shim needed; the module namespace stays `spare_paw.tui`.

## Risks

- **Streaming + tool calls is subtle.** OpenRouter streams tool-call deltas with incremental JSON args; assembly logic must be correct or tool calls silently break. Mitigation: dedicated unit tests for `chat_stream` parsing and `_stream_and_assemble` reconstruction.
- **Per-turn widget count growth.** Long sessions with many turns will accumulate `MessageView` widgets. Textual handles this fine up to thousands, but eventual virtualization may be needed. Out of scope for this spec.
- **Token coalescing window.** Too short → wasted CPU; too long → laggy feel. Default 16ms, make configurable in `tui/theme.py` if needed.
